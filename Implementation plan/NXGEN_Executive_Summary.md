### Common Pitfalls to Avoid
❌ Starting without feature list validation  
❌ Unclear role separation (Admin/Operator/Installer)  
❌ No ongoing maintenance plan  
❌ Insufficient SME review time  
❌ Trying to document everything at once  
❌ No user testing before launch  

### Best Practices
✅ Phase approach: Core content first, nice-to-have later  
✅ Template-driven: Consistency through standardization  
✅ User-centric: Organize by task, not system architecture  
✅ Living documentation: CMS enables continuous improvement  
✅ Feedback loops: User comments, ticket analysis, analytics  
✅ Cross-linking: Connect related content extensively

## 💡 Key Insights from Working Session

From your transcript, several critical insights were captured:

1. **Role-Based Documentation is Essential**
   - Customers like VLG (admin-only) vs C-Contact (admin + installer)
   - Solution: Separate guides with clear linking between them

2. **Installer Access Challenge**
   - Many installers don't have GCXONE access
   - Need downloadable PDFs if docs are private
   - Or make installer-specific content public

3. **Dashboard Should Show "Why" Not Just "What"**
   - Focus on utility and problem-solving
   - Example: "Active Sites widget shows which sites send too many alarms"
   - Users need to understand value, not just features

4. **Talos Documentation Scope**
   - Only platform essentials (token setup, configuration)
   - NOT deep Evalink internals
   - Customers configure what they need, we handle backend

5. **Features Need Configuration + Overview**
   - Feature overview in Features section
   - How to configure in Admin Guide
   - These must be tightly linked

6. **Prerequisites Are Critical**
   - Use existing starter pack as foundation
   - IPs, ports, whitelisting, software requirements
   - Front-load this information

These insights are incorporated throughout the architecture.
