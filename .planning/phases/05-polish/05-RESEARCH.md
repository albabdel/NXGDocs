## Summary

Phase 5 is gated on Phase 4 (Content Migration). Once content lives in Sanity and is rendered via `.sanity-cache/`, two concrete problems need solving: (1) Algolia search is currently broken in a specific way ‚Äî the `docusaurus.config.ts` references `indexName: "Documentation site"` while the actual populated index (used by the custom indexer script) is named `nxgen_docs` with a different App ID (`0QV3FAFAD5` vs `V5T3AW2AU9`). This mismatch must be resolved before any post-migration re-crawl will return valid results. (2) The CSS has 3,531 lines and works well in dark mode but shows known light-mode breakage: section headings in `index.tsx` use `text-[#E8B058]` (too low contrast on white backgrounds) and the VoC modal success message uses `rgba(255,255,255,0.7)` which is invisible in light mode. Beyond those two targeted fixes, the design system is well-tokenized and should not be changed wholesale.

Navigation is structurally sound but the sidebar serving Sanity content (`sanity-docs` plugin instance) has `sidebarPath: undefined` (auto-generated) and will need `sidebar_position` frontmatter set in the Sanity content to produce a logical order after migration. Footer links pointing to `#` and `/docs` with vague labels (`Devices`, `Features`) are a UX problem but are low effort to fix.

**Primary recommendation:** Resolve the Algolia credential/index name conflict, trigger a fresh crawl after Phase 4, fix the two known CSS light-mode issues, and set `sidebar_position` in Sanity document frontmatter during migration.

### Pattern 1: Algolia contextualSearch with auto-generated sidebar

The current config enables `contextualSearch: true`. This means Algolia filters results by `docusaurus_tag`, `language`, `lang`, and `version` facets. The `sanity-docs` plugin instance uses `routeBasePath: 'docs'` ‚Äî Docusaurus will tag these pages as `docs-sanity-docs-current`. The Algolia crawler must be allowed to index `docusaurus_tag` as a faceted attribute or the contextual filter silently returns zero results.

**What:** After Phase 4 migration, the Algolia crawl must be triggered manually from the Algolia dashboard. The crawler must have `renderJavaScript: true` because Docusaurus is a React SPA.

**When to use:** After every major content structure change.

### Pattern 2: Auto-generated sidebar ordering via frontmatter

The `sanity-docs` plugin uses `sidebarPath: undefined` (auto-generated from filesystem). Docusaurus reads `sidebar_position` frontmatter from each `.md` file. The `docusaurus-plugin-sanity-content` already writes `sidebar_position: ${doc.sidebarPosition}` if the field is set in Sanity. The Sanity `doc` schema has a `sidebarPosition` field. Ordering is controlled by setting that field in Sanity during Phase 4 migration.

**Example (existing pattern in `plugins/docusaurus-plugin-sanity-content/index.js`):**
```js
// Source: classic/plugins/docusaurus-plugin-sanity-content/index.js line 135
if (doc.sidebarPosition != null) {
  fm += `sidebar_position: ${doc.sidebarPosition}\n`;
}
```

### Pattern 3: CSS token-first styling for landing pages

The CSS has two layers:
1. CSS custom properties defined in `:root` and `[data-theme='dark']`
2. Tailwind utility classes overridden for light mode in the `LIGHT MODE OVERRIDES` section (lines ~1690-1800 of custom.css)

The correct pattern for brand color in JSX is:
```tsx
// Use CSS token ‚Äî not hardcoded hex
<h2 style={{ color: 'var(--ifm-color-primary)' }}>Section Title</h2>
// OR use the WCAG-safe gold token explicitly
<h2 className="text-themed-gold">Section Title</h2>
```

The current violation in `index.tsx`:
```tsx
// WRONG: hardcoded #E8B058 is invisible on white in light mode
<h2 className="text-3xl font-bold text-[#E8B058]">Quick Start</h2>

// CORRECT: uses existing CSS override infrastructure
<h2 className="text-3xl font-bold text-[#E8B058]">Quick Start</h2>
// ... works because [data-theme='light'] .text-[\#E8B058] { color: #996B1F !important; }
// is already defined in custom.css at line ~1753
```

Wait ‚Äî this override IS already present in custom.css. The heading issue may be already handled. The real violation to verify is the VoC modal:

```css
/* custom.css line ~2856 ‚Äî hardcoded white, no light-mode override */
.voc-success-message p {
  font-size: 1.1rem;
  color: rgba(255, 255, 255, 0.7);  /* invisible in light mode */
  max-width: 80%;
}
```

### Anti-Patterns to Avoid

- **Re-indexing manually from files**: The custom `index-documentation.js` script reads from `classic/docs/` but after Phase 4, content lives in `.sanity-cache/docs/`. Do not run the old script against the wrong directory.
- **Deleting and re-creating the Algolia index by hand**: After migration, use the Algolia Crawler dashboard to trigger a new crawl against the live URL. The crawler handles `docusaurus_tag` injection that the custom script does not.
- **Changing the overall visual design**: The CSS `[data-theme='dark']` block is marked `IMMUTABLE: These values match the current "Golden Master" dark theme.` Do not touch dark mode colors.

## Common Pitfalls

### Pitfall 1: Algolia credential/index name split-brain

**What goes wrong:** The site uses `appId: V5T3AW2AU9` and `indexName: "Documentation site"` in `docusaurus.config.ts` (what DocSearch uses for search queries). But the custom indexer script (`scripts/index-documentation.js`) uses `appId: 0QV3FAFAD5` and `indexName: nxgen_docs`. These are different Algolia applications. DocSearch searches the first; the custom script writes to the second. Users see empty or stale results.

**Why it happens:** The project had two sets of credentials from an earlier migration. The credentials file (`ALGOLIA_CREDENTIALS_CHECK.md`) acknowledges both sets exist. No one resolved which is canonical.

**How to avoid:** Before triggering any crawl, decide which App ID is the live production one. Check the Algolia dashboard to see which index has actual data. Update `docusaurus.config.ts` to match.

**Warning signs:** Search returns "No results" or shows 404 URLs for content that exists.

### Pitfall 2: contextualSearch returns zero results for Sanity content

**What goes wrong:** After Phase 4, the `sanity-docs` plugin instance routes content to `/docs/`. Algolia's `contextualSearch: true` creates a facet filter for `docusaurus_tag:docs-sanity-docs-current`. If the crawler did not inject this tag (or injected the wrong one), search returns nothing for `/docs/` pages.

**Why it happens:** The crawler uses JavaScript rendering to extract the `docusaurus_tag` meta content that Docusaurus injects into each page. If `renderJavaScript: false` in the crawler config, this metadata is never captured.

**How to avoid:** Ensure the Algolia crawler has `renderJavaScript: true`. Trigger a full re-crawl after Phase 4 is complete. Verify with the Algolia dashboard that records exist with the `docs-sanity-docs-current` tag.

**Warning signs:** Search works on the home page but not on `/docs/` pages.

### Pitfall 3: VoC modal invisible text in light mode

**What goes wrong:** `.voc-success-message p { color: rgba(255, 255, 255, 0.7); }` is hardcoded white with no light-mode override in custom.css.

**Why it happens:** The VoC modal was designed for dark mode and a light-mode override was never added to the CSS.

**How to avoid:** Add `[data-theme='light'] .voc-success-message p { color: var(--text-secondary); }` to custom.css.

**Warning signs:** In light mode, the success confirmation message after submitting feedback is blank/invisible.

### Pitfall 4: Docusaurus 3.9 / DocSearch v4 compatibility

**What goes wrong:** The current package is `@docusaurus/preset-classic: ^3.9.2`. Docusaurus 3.9 added DocSearch v4 support. If the site is running 3.9 and DocSearch was upgraded silently, the `DocSearch-*` CSS class names may have changed (some were renamed in v4).

**Why it happens:** The CSS in `custom.css` lines 2956-3531 targets `DocSearch-Button`, `DocSearch-Modal`, etc. If DocSearch v4 changed class names, the custom styling breaks.

**How to avoid:** Check the current `@docsearch/react` version in `node_modules`. Docusaurus 3.9 ships with v4 support but keeps v3 as default ‚Äî the theme CSS targets v3 class names unless the site explicitly opts into v4.

**Warning signs:** Search button loses custom NXGEN gold styling after a dependency update.

   - What we know: Both exist in the codebase. `V5T3AW2AU9` is in `docusaurus.config.ts` (used by DocSearch UI). `0QV3FAFAD5` is in `scripts/index-documentation.js` and `algolia-config.json` (used by the custom indexer).
   - What's unclear: Which Algolia application has an active DocSearch subscription and a real index populated from the live site.
   - Recommendation: Log into the Algolia dashboard before Phase 5 work begins. Identify which app ID has a non-empty index. Update both `docusaurus.config.ts` and the credential files to reference only one app.

2. **Does the Algolia crawler have `renderJavaScript: true` in the current config?**
   - What we know: The crawler config accessible from the Algolia dashboard (not committed to the repo) controls this. Docusaurus is a React SPA.
   - What's unclear: Whether the crawler was ever configured with JS rendering enabled.
   - Recommendation: Verify from the Algolia Crawler dashboard. Enable `renderJavaScript: true` if not set.

3. **Will Phase 4 keep any MDX files in `classic/docs/` or will it be fully empty?**
   - What we know: Phase 4 migrates all docs to Sanity. The `sanity-docs` plugin reads from `.sanity-cache/docs/`. The role-based docs instances (`docs-admin`, `docs-manager`, etc.) read from `docs-admin/`, `docs-manager/`, etc.
   - What's unclear: After Phase 4, are the role-specific subdirectories also emptied into Sanity, or do they remain as static MDX?
   - Recommendation: The sidebar refinement plan should be deferred until Phase 4 clarifies which directories remain. The `sanity-docs` sidebar is auto-generated from `.sanity-cache/docs/` and controlled purely by `sidebar_position` frontmatter.
